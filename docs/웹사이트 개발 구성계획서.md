# 1) 목표/범위 정의 (Scope)

## 1-1. 목표

- 브랜드 감성 중심의 쇼핑몰 구축 (미니멀 UI + 고화질 이미지 + 룩북/저널 중심)
- 구매 전환 흐름 단순화 (카테고리 → 리스트 → 상세 → 장바구니 → 결제)
- 운영자가 혼자서도 콘텐츠/상품/주문을 처리 가능한 관리자 기능 제공
## 1-2. 범위 (MVP)

- Shop(카테고리/리스트/상세)
- Cart/Checkout
- Login/MyPage(주문조회)
- Notice/FAQ/Q&A
- Lookbook/Journal(콘텐츠 게시)

> 참고 사이트의 대표 메뉴/구성:
> 
> - ESPIONAGE: ALL ITEM/OUTER/TOP/PANTS/ACC + LOOKBOOK/JOURNAL/NEWS + FAQ/Q&A + LOGIN/CART/MYPAGE
>     
> - DOFFJASON: Shop(전체/신상품/아울렛/레더/아우터/상의/하의/악세서리) + COLLECTION + NOTICE/REVIEW/FAQ/PRESS/JOURNAL

# 7) 기술 스택 제안 (1인 + 멀티에이전트에 최적)

**권장(브랜드형 커스텀/SEO/속도 유리)**

- Front: Next.js(SSR/SEO) + Tailwind
- Back: Next.js API Routes 또는 NestJS(확장 시)
- DB: PostgreSQL + Prisma
- 이미지: S3 호환 스토리지 + CDN(CloudFront 등)
- 인증: NextAuth(또는 자체 JWT)
- 결제: 토스페이먼츠/아임포트 등

> 이유: 브랜드형 쇼핑몰은 이미지/SEO/퍼포먼스가 중요해서 “Next 기반”이 운영에 유리합니다.

# 9) 멀티에이전트 개발 운영 구조 (1인 개발 최적)

## 9-1. 에이전트 역할 분리(권장 6개)

1. **PM/Architect(오케스트레이터)**
- 요구사항, 범위관리, IA/컴포넌트 기준, PR 리뷰 기준
    

2. **UI/UX Agent**
    

- 레이아웃/스타일 가이드, 컴포넌트 설계, 반응형
    

3. **Frontend Agent**
    

- Shop/List/Detail/Cart/Checkout 구현
    

4. **Backend/Data Agent**
    

- DB/Prisma 모델, API 설계, 관리자 기능
    

5. **Commerce/Payment Agent**
    

- 주문/결제 플로우, 결제사 연동, 웹훅/상태머신
    

6. **QA/Release Agent**
    

- 테스트 시나리오, 체크리스트, 배포/릴리즈 노트
    

## 9-2. 작업 운영 규칙(혼자 + 멀티에이전트 핵심)

- “한 번에 한 에이전트 = 한 PR 단위”로 쪼개기
    
- 공통 SSOT 문서 3개를 최상위에 둠
    
    - `PRD.md` (요구사항/범위)
        
    - `IA.md` (메뉴/페이지 구조)
        
    - `API.md` (엔드포인트/DTO)
        

## 9-3. tmux/폴더 구조 예시

- tmux 4~6 pane
    
    - Pane1: Orchestrator(계획/이슈/PR 리뷰)
        
    - Pane2: Frontend 작업
        
    - Pane3: Backend 작업
        
    - Pane4: QA/테스트/리그레션
        
- repo 예시
    
    - `/docs` : PRD/IA/API/DB/UX
        
    - `/apps/web` : Next 프론트
        
    - `/apps/admin` : (선택) 관리자 별도
        
    - `/packages/ui` : 공용 컴포넌트
        
    - `/packages/db` : Prisma 스키마/마이그레이션
